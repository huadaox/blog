# 为什么当前 LLM 难以独立完成高一致性的长篇小说

在没有人工参与的情况下，只靠当前主流 LLM 稳定写出十万字以上、长期保持高一致性的小说，极其困难。

这不是"当前模型还不够好"的问题，也不是提示词写得不够精妙的问题。它指向的是当前主流自回归语言模型范式下的一类结构性限制。

写长篇小说的核心挑战，不在于遣词造句——这恰恰是 LLM 最擅长的事——而在于维持一个不断演化的世界状态。角色的性格弧线要在几十万字中保持连贯，情节的因果链条要前后自洽，第五章埋下的伏笔要在第三十章被自然地呼应。这些要求的本质，是对一个复杂状态的长期精确维护。而当前主流 LLM 恰恰缺乏一个可工程化、可持久维护的任务级世界状态机制。

## 当前的主流做法

由于 LLM 单次输出长度有限，长篇小说只能逐章节生成。每生成一个新章节，都需要向模型提供此前故事的背景信息作为上下文。围绕这个基本约束，目前主要有三种技术路线。

**路线一：逐层细化大纲。** 从一个高层故事构想出发，先生成粗纲，再拆分为细纲，最后逐章展开为正文。这条路线的问题在于：写手在展开某一章时，只能看到局部的细纲片段，无法看到故事的全貌。大纲提供的是结构骨架，但小说的血肉——角色的微妙变化、场景之间的情感承接、前文中一个不经意的细节在后文中的回响——这些东西存在于骨架之外。细纲里写着"李明在本章与张华产生矛盾"，但矛盾的质感、分寸、以及它和前文十几个场景之间的情感关联，不是一行大纲能承载的。

**路线二：多 Agent 角色分工。** 设立多个专门化的 Agent——世界观构建者、导演、写手、审查员——各司其职，协同完成创作。这条路线把问题转化成了一个组织架构问题，看起来很合理。但核心瓶颈暴露在审查员身上：它需要维持一个精确的故事事实状态，来检查新生成的内容是否与已有设定矛盾。而审查员本身也是一个 LLM，它面临与写手完全相同的困境——无法可靠地追踪一个在十万字中不断变化的世界。让一个有同样缺陷的模型去审查另一个模型，只是把问题推迟了，没有消除它。

**路线三：模拟世界。** 构建一个由多个 Agent 扮演角色的虚拟小镇，让角色在模拟环境中自主行动、对话、产生事件，然后将发生的事件转写为小说文本。这是最有想象力的方案，但也逃不开同样的困局：每个 Agent 的记忆和决策都由 LLM 驱动，长期运行后，角色行为不可避免地偏离初始设定。那个"安静内向的图书管理员"可能在第 30 天突然变成了社交达人，而系统中没有任何机制能察觉这种漂移。

三条路线，架构各异，但都在同一个地方卡住了：**无法构建一个可靠的、可供 LLM 稳定使用的任务级叙事状态。**

## 实践中的真实困境

上面的分析还比较抽象。下面这些问题更多来自实践层面的稳定观察：如果你真正动手做过 AI 长篇写作，它们大概率都不会陌生。

**显式提示的锚定效应。** 当你在上下文中提供详细的背景信息——角色关系、未解决的冲突、需要推进的情节线——模型的行为会发生一个微妙但致命的变化：它开始"完成任务"而非"讲故事"。你写明"本章需要体现李明对张华的不信任"，模型就会在每一段对话、每一个动作里塞进不信任的信号，像一个过于用力的演员。它不是在自然地展开一个场景，而是在逐条核销你提供的信息清单。提供的背景越具体，这种锚定越严重——模型的注意力被显式信息吸走，丧失了自然展开、制造意外、留白和暗示的能力。

**角色塑造的两难困境。** 在提示词中显式描述角色性格（"王芳：性格倔强，重情义，说话直接"），模型就会机械地执行这些标签。每句对话都在喊"我很倔强"，角色成了脸谱。这不是活生生的人物，而是一组属性标签的拟人化。但如果改用隐式的方式——不直接描述性格，而是通过前文的行为和对话让模型"理解"角色——情况也好不到哪里去。跨几个章节之后，没有显式锚点的性格就开始漂移。上一章还沉默寡言的人，这一章突然变得话多起来，因为模型已经"忘了"这个角色是谁。显式描述导致死板，隐式暗示导致漂移，两头都走不通。

**章节的模板化。** 还有一个更隐蔽的问题：由于模型的单次输出长度有限，每个章节的生成是一次独立的请求。而 LLM 有一个根深蒂固的倾向——把每次输出写成一个结构完整的小故事，有起承转合，有情感闭环。十个章节读下来，每一章都像一篇独立的短篇小说，有着几乎相同的叙事节奏。长篇小说应有的那种跨章节的张力蓄积、节奏变化、留白与爆发的交替，在这种逐章生成的模式下几乎不可能实现。

这些问题看起来可以通过更精巧的提示工程来缓解。单独拿出任何一个，确实可以通过特定的 workaround 部分改善。但当你试图同时解决所有问题时，会发现它们此消彼长——解决了角色脸谱化，就加剧了性格漂移；试图打破章节的模板化，又引入了新的信息锚定问题。这种按下葫芦浮起瓢的困局不是巧合，而是同一个深层原因的不同症状。

## 根本原因

上述这些现象——范式层面的失败和实践中的具体困境——可以归结为三个相互关联、层层递进的技术原因。

### 上下文腐烂

LLM 处理信息的唯一通道是文本上下文。无论是直接拼接前文、还是通过 RAG 检索相关片段、还是用摘要压缩历史内容，最终都要以文本的形式进入模型的注意力窗口。

但上下文不是一个可靠的信息通道。当上下文变长，模型对信息的提取能力呈现出一种被称为"U 型崩溃"的模式：对开头和结尾的信息保持较好的注意力，而对中间大量内容的处理能力急剧下降。Liu et al. (2023) 的论文 *Lost in the Middle: How Language Models Use Long Contexts* 系统地验证了这一点。

这意味着什么？当你把一部小说的前文信息塞入上下文——角色关系的变化、早期埋下的伏笔、次要情节线的进展——其中大量内容实质上处于模型的"盲区"。模型不是选择性地忽略了这些信息，而是在注意力机制层面就没有有效地处理它们。

更糟糕的是，这个问题在压缩后不但没有消失，反而换了一种形式存在。为了应对上下文长度限制，常见的做法是将前文压缩为摘要。但摘要本身就是一次有损压缩——它必然丢失细节。哪些细节被保留、哪些被丢弃，取决于执行摘要的 LLM 在那个时刻的"判断"。而这个判断本身就可能是有偏的、遗漏的。一个在第三章看起来不重要的细节，可能在第二十章成为关键情节转折的基础。但它在早期的摘要中已经被丢弃了，永久地从系统的"记忆"中消失。

上下文越长，信息通道越不可靠。你往里塞的信息越多，能被准确提取的比例越低。

### 误差的滚雪球

如果说上下文腐烂是信息传递的损耗，那么误差累积就是这种损耗的动态后果。

自回归模型的生成方式是逐 Token 预测的：每一个新 Token 都基于前面所有 Token 生成。这意味着每一步都有微小的错误概率。Yann LeCun 近年反复用一个简单的数学估算来说明长程自回归生成的脆弱性：如果每个 Token 的错误概率为 $e$，生成 $N$ 个 Token 后整体正确的概率为 $(1-e)^N$。对于十万字级别的文本——大约 15 到 20 万 Token——即使 $e$ 极小，这个值也会快速下降。这当然不是对小说生成的严格证明，但它直观地说明了：当生成链条足够长，微小偏差的累积会变得不可忽视。

但真正的问题不是错误会发生，而是错误会自我强化。Zhang et al. (2023) 在 *How Language Model Hallucinations Can Snowball* 中揭示了一个关键机制：当 LLM 在前文犯下一个微小的事实错误后，它不会在后续生成中自动修正这个错误。相反，为了维持上下文的表面连贯性，模型会倾向于与前文的错误保持一致——实质上是在"圆谎"。一个小错误引出一个与之配套的新细节，新细节又引出更多偏离事实的内容。这篇工作讨论的主要不是小说，但它为长篇叙事中的误差级联提供了一个重要旁证。

这就解释了为什么 AI 生成的长篇小说往往呈现出一种特征性的退化曲线：前几章质量尚可，中段开始出现轻微的不一致，到后期则完全失控。不是模型在后半段变蠢了，而是前半段累积的微小偏差，经过十几万 Token 的传递和放大，已经严重到了无法忽视的程度。

### 缺乏外部纠错机制

误差会累积，那能不能及时纠正？这个问题把我们引向了第三个、也是最根本的原因。

在代码生成任务中，LLM 有一个巨大的结构性优势：编译器和测试用例提供了硬性的外部反馈。代码要么能跑，要么不能跑；测试要么通过，要么不通过。这种二值化的反馈信号与模型本身完全独立，能够有效地截断误差的累积链条。即使模型犯了错，编译失败或测试不通过的信号会迫使系统修正，阻止错误向下游蔓延。

小说没有编译器。

那么，能不能用另一个 LLM 来充当"审查员"，检查生成内容与已有设定之间的一致性？这是一个自然的想法，也正是多 Agent 方案中"审查员"角色的设计初衷。Kambhampati (2024) 在 ICML 论文 *LLMs Can't Plan, But Can Help Planning in LLM-Modulo Frameworks* 中并不是直接证明"小说审查必然失败"，但他的论证为这种怀疑提供了重要支撑：如果一个系统缺乏独立于 LLM 的外部验证器，那么单靠 LLM 之间互相审查，很难形成稳定的负反馈闭环。

用 LLM 审查 LLM 的生成内容，相当于用一个有同样认知缺陷的系统去校验另一个系统。审查 LLM 同样会受到上下文腐烂的影响，同样会在长文本中丢失关键信息，同样会被表面的连贯性欺骗。它可以发现一些明显的矛盾（角色在同一章里名字前后不一致），但对于那些需要跨越几十章、追踪多条情节线才能发现的深层不一致——比如角色动机的悄然转变、因果链条的断裂——它的检测能力并不比生成模型本身强多少。

代码世界有编译器作为独立于模型的客观裁判。语义世界里没有这样的裁判。没有一个外部系统能自动判定"这个角色在第 37 章的行为是否与第 5 章的性格设定一致"。在当前的技术条件下，能较可靠地做出这种判断的，仍然主要是人类读者。

三层原因叠加在一起，构成了一个闭合的困局：上下文腐烂导致信息传递有损，有损的信息引发误差累积，累积的误差又没有外部机制能纠正。这不是某一个环节的技术缺陷，而是当前范式下的结构性限制。

## 缺失的那块拼图

综合以上分析，可以得出一个更精确的判断：

在当前主流 LLM 范式下，如果一个长篇小说生成系统主要依赖 LLM 通过文本上下文来读取、压缩、更新和执行叙事状态，那么该系统的长期一致性控制能力会随着文本长度增长而系统性退化。长上下文窗口、外部记忆检索、LLM 自我修订，这些技术手段可以延缓退化的速度，但很难从根本上阻止退化的发生。

问题的根源在于：这些系统缺少一个**可工程化的、可更新的任务级世界状态机制**——就像人类作家在写作时脑海中自然维持的那个东西。

一个小说家在创作长篇时，脑子里始终运转着一个关于故事世界的内部模型。这个模型不是一份清单或数据库，它更像是一种直觉式的、整体性的把握：角色之间的张力、情节的走向、哪些线索已经埋下但尚未兑现、此刻的情感基调应该是什么。它随着每一章的写作而自然更新，是一切叙事决策的隐性基础。

谢赛宁在描述世界模型时给出过一个精确的定义：给定一个系统的当前状态 $s_t$ 和一个动作 $a_t$，世界模型是一个能预测下一个状态 $s_{t+1}$ 的转移函数。这个状态是系统的充分统计量——它保留对未来预测和决策所需的关键信息，同时忽略无关细节。而构建这样的状态，本质上是一个表征学习问题。

关键在于：这个状态不一定是显式的、结构化的。它可以是特征空间中的一个隐藏表示（latent representation），可以是层次化的、抽象的。就像我们不需要模拟每一个空气分子的碰撞来理解天气，故事的世界状态也不需要把每一个细节都用自然语言写出来。它需要的是在某个表征空间中，充分地编码故事世界的关键信息——角色关系、情节因果、未解决的冲突、情感张力——并且随着故事的发展而持续更新。

这里需要说得更精确一些。LLM 并不是完全没有世界知识；它显然学到了大量关于现实与叙事的统计规律。问题在于，当前主流 LLM 缺乏一个可持久、可读写、可验证、适合长程任务控制的世界状态机制。它对故事世界的理解更多散落在文本上下文和隐式表征中，每次生成都要从一堆文本片段中临时重建局部状态。这不是一个稳定可操作的状态层，而是一种一次性的、有损的、随时可能走样的状态重建过程。上下文窗口里的信息可能被忽视，检索回来的段落可能不完整，压缩摘要必然丢失细节，LLM 的自我修订可能引入新的错误。缺少的不是任何意义上的"世界知识"，而是一个能被稳定维护和持续更新的任务级世界状态接口。

## 结语

笔者在过去相当长的时间里反复尝试过各种流程编排——调整大纲的粒度、改进摘要的策略、增加审查环节、在提示词上做各种精细的控制。上面提到的那些问题，每一个都真实地、反复地出现过。很长一段时间里，我以为问题的核心在记忆的压缩、保存和检索上——如果能让模型更准确地"记住"前文，是不是一切就解决了？

后来逐渐意识到，记忆压缩的失真、检索的遗漏、角色的漂移、章节的模板化，这些看似不同的问题其实指向同一个缺失。真正让我把这个想法想清楚的，是谢赛宁对世界模型的描述：一个模型应该拥有一个可更新的世界状态，这个状态是系统的充分统计量，包含对未来预测和决策所需的关键信息，并且可以在特征空间中以隐藏表征的形式存在。这和我在小说生成中反复碰壁后得出的直觉完全吻合——小说生成系统缺少的核心东西，不是更好的记忆检索，而是一个能随故事发展持续更新的世界状态机制。更准确地说，当前 LLM 也许拥有隐式的世界知识，但没有一个可工程化的长期叙事状态层。

这不是一个靠增大上下文窗口或设计更巧妙的 Agent 流程就能轻易解决的问题。当前的 LLM 把故事的"状态"主要寄托在文本上下文上——这相当于一个作家每写一章都要靠前文片段、摘要和提示词来重新回忆故事世界。这种方式可以工作一段时间，但随着篇幅增长，关键而低频的叙事状态就越来越容易丢失、走样、漂移。

也许解决方案的方向，在于让模型真正拥有一个可更新的内部世界状态——不一定是显式的数据库或知识图谱，也可以是潜在空间中的某种表征——但它必须能随着叙事的推进而持续演化，能编码故事世界中真正重要的信息，并且能可靠地指导后续的生成。
